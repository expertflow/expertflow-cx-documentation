# WebRTC Detail Report

The **WebRTC Detail Report** shows a row-level record for each WebRTC link generated by the platform. Use it to diagnose individual call failures, investigate expiration patterns, and audit WebRTC link usage per agent and channel.

## Report Summary

| Attribute | Detail |
|---|---|
| **Report type** | Historical |
| **Primary audience** | Solution Admin, Supervisor |
| **Key use case** | Diagnose WebRTC connection failures; audit link generation and expiration |

## Report Columns

| Field | Description |
|---|---|
| **Conversation ID** | Unique identifier of the customer conversation for which the WebRTC link was generated. |
| **Agent Name** | Name of the agent who participated in the conversation. |
| **Status** | Outcome of the WebRTC link. Possible values: `successful`, `unsuccessful`, `state_changed`, `expired`. |
| **Expiration Reason** | Reason the WebRTC link terminated. Possible values: `successful_call`, `not_answered`, `timeout`. |
| **Channel** | The channel associated with this conversation (e.g., voice, web). |
| **Link Generation Time** | Timestamp when the WebRTC link was generated. |

## Report Filters

| Filter | Description |
|---|---|
| **DateTime** | Filter by date and time range to scope the report to a specific period. |

## How to Read This Report

Focus on rows with **Status: unsuccessful** or **Expiration Reason: not_answered** to identify where WebRTC connections are failing. Group by **Agent Name** to determine whether failures are concentrated around specific agents (suggesting a device or browser configuration issue) or spread evenly (suggesting a platform or network issue).

A high **timeout** expiration reason rate indicates that generated links are not being opened by customers within the validity window — consider reviewing your link delivery mechanism or expiry duration.
