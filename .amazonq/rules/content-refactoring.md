# ExpertFlow CX Docs ‚Äî Content Refactoring Rules

These rules are derived from Phase3 project decisions (`Restructure_Project_Plan.md`, `Final_12_Persona_Model.md`) and the refactoring work done on the `Restructured/` content tree.

---

## 1. Section Placement by doc-type (Diataxis)

| `doc-type` | Belongs in |
|---|---|
| `how-to` | `How-to_Guides/<Persona>/` |
| `explanation` | `Capabilities/<Area>/` |
| `reference` | `Capabilities/<Area>/` or `Reference/` (APIs, schemas, specs) |
| `tutorial` | `Getting_Started/For_<Persona>/` |
| `landing` | Index file of its own section |

**`Capabilities/` is for understanding what the platform does ‚Äî not for step-by-step instructions.**
Any file with `doc-type: how-to` found inside `Capabilities/` must be moved to the correct `How-to_Guides/<Persona>/` folder.

---

## 2. Persona Mapping

When moving a `how-to` file, use this table to determine the target persona folder:

| Content type | Target folder |
|---|---|
| Channel configuration (email, Facebook, LinkedIn, Twitter, widget‚Ä¶) | `Administrator` |
| Routing, queues, agent desk, user management | `Administrator` |
| Reporting setup, Superset configuration | `Administrator` |
| Agent-facing procedures | `Agent` |
| Supervisor dashboards, QA workflows, WFM scheduling | `Supervisor_and_QA_Lead` |
| Installation, upgrades, RKE2/Helm/SSL/networking, monitoring, backup | `Platform_Operator` |
| Tenant onboarding, tenant lifecycle, subscriptions, billing, white-labeling | `Partner` |
| SDK, API, bot connector, CRM integration, embedding | `Developer_Integrator` |
| Conversation Studio, IVR flows, AI/NLU | `Conversation_Designer` |

---

## 3. Audience Metadata Tag Schema

When updating or creating frontmatter, use these canonical `audience` values:

```
agent | supervisor-qa | administrator | platform-operator | partner |
conversation-designer | developer-integrator | platform-overview
```

- `platform-operator` is deployment-agnostic ‚Äî use for any installation, upgrade, monitoring, alerting, backup, or infrastructure content regardless of single-tenant or multi-tenant context
- `partner` covers all tenant management and commercial content ‚Äî onboarding, lifecycle, subscriptions, billing, white-labeling
- `hosting-partner` is retired ‚Äî replace with `platform-operator` and/or `partner` as appropriate
- `platform-overview` files have no persona golden path
- Pure `reference` files carry no `audience` tag (accessed via Reference nav + search)

---

## 4. Index Page Format

All `Capabilities/<Area>/index.md` files must use **structured tables**, not flat bullet lists.

Follow the `Digital_Channels/index.md` pattern:
- Group content by logical sub-topic
- Use columns: topic/feature | overview link | configuration/how-to link | notes/limitations link
- End with a `## Related` section linking to adjacent sections

---

## 5. Sidebar Rules

- `How-to_Guides/<Persona>/` folders use `{ type: 'autogenerated' }` ‚Äî moving a file there is sufficient, no manual sidebar entry needed.
- When a file moves **out of** `Capabilities/`, remove its explicit sidebar entry from the Capabilities section in `sidebars.js`.
- When a file moves **into** a manually-managed sidebar section, add an explicit entry.
- The sidebar doc id must match the new file path exactly (e.g. `cx/How-to_Guides/Developer_Integrator/My-File`).

---

## 6. Cross-link Updates (mandatory after every move)

After moving any file:

1. Run: `grep -rl "<filename>" /path/to/Restructured/` to find all referencing files.
2. Update every relative path in those files to reflect the new location.
3. Also update the moved file's own internal links (its "Related Articles" section), since relative paths change when the file moves.
4. Check `sidebars.js` for any explicit reference to the old path.

---

## 7. Page Title Consistency

- Page titles (`# H1`) must match the `title:` frontmatter field.
- Titles should be concise ‚Äî avoid redundant suffixes like "Features and Capabilities" or "Configuration Guide" when the context is already clear from the section.
- Link label text in other files should match the page's `title:` field.

---

## 8. File Naming Convention

- Use kebab-case: `My-Feature-Guide.md`
- File name should match the page title closely but does not need to be identical
- Do not rename files without also updating all cross-links and the sidebar
