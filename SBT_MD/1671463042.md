# CX Knowledgebase : Two-Factor Authentication (2FA) – Business Overview

As security landscapes evolve and regulatory requirements for identity verification tighten, Two-Factor Authentication (2FA) has become a critical component of access management. The EFCX 2FA Service provides a robust, tenant-wide security layer that mitigates unauthorized access risks by requiring a secondary verification step during agent login.  
  
This capability is designed to be **configurable, compliant, and cost-effective** , allowing organizations to protect their applications using modern, flexible verification methods—including **Authenticator Apps, SMS, and Email and easy support for integration with other channels like whatsapp** —without altering core application logic.

### Business Value

  * **Enhanced Security Posture:** Mitigates risks of credential compromise (phishing, weak passwords) and SIM-swapping attacks (via Authenticator Apps).

  * **Regulatory Compliance:** assists organizations in meeting industry standards (e.g., PCI-DSS, SOC2) that mandate multi-factor authentication for access control.

  * **Operational Flexibility:** offers configurable delivery channels (SMS or Email) to balance security needs with operational costs and user preference.




* * *

## Key Concepts

### Tenant-Wide Scope

The 2FA service operates at the **Tenant Level**. It is a global setting; once enabled, the security policy applies uniformly to all users and all EFCX applications within that tenant. This ensures consistent security enforcement across the organization, preventing coverage gaps.

### Flexible Verification Channels

Tenants can select the verification method that best aligns with their security requirements and budget. The service supports:

**Channel**| **Description**| **Best For**| **Cost Impact**  
---|---|---|---  
**SMS**|  Delivers OTP to the agent’s mobile device via standard text messaging.| Standard Use Cases| High (Per Message)  
**Email**|  Delivers OTP to the agent’s corporate email address.| Ease of Deployment| Low / None  
**Authenticator App (TOTP)**  
**(Future Integration)**|  Uses apps like **Google Authenticator** or **Microsoft Authenticator** to generate rotating codes. Works **offline** without cellular signal.| High Security & Zero Cost| **None**  
**WhatsApp (Future Integration)**|  Delivers OTPs via the agent’s WhatsApp account. Encrypted and highly reliable in international markets.| Global Teams & High Reliability| Varies (Provider)  
  
### Dynamic Contact Registration

The service eliminates administrative overhead by supporting **Just-in-Time (JIT) Registration**. Agents are not required to pre-register manually. The system detects missing 2FA setups during the first login and guides the agent through a mandatory configuration flow specific to the chosen channel.

## Administrative Configuration

The 2FA service is **disabled by default**. A Tenant Administrator must explicitly configure the capability via **2FA Configurations** in **Tenant Settings** module.

### Configuration Workflow

  1. **Service Activation:** The administrator toggles the service status to "Enabled."

  2. **Channel Selection:** The administrator selects the **Single Preferred Channel** for the tenant (e.g., _Authenticator App_ OR _WhatsApp_).

     * _Note: To ensure consistent behavior and simplified support, only one channel type is active per tenant at a time._

  3. **Provider Assignment:**

     * For **Email/SMS** : Select the specific channel.

     * For **Authenticator Apps** : No external provider is needed; the system handles key generation natively.

  4. **Security Policy Definition:**

     * **OTP Validity:** Duration (in minutes) a code remains valid.

     * **Max Retry Limits:** Threshold for invalid attempts before triggering a temporary security lockout.

## 4\. User Experience (UX)

The workflow adapts automatically based on the configured channel.

### The Login Flow

  1. **Primary Authentication:** Agent enters username and password.

  2. **Verification Challenge:** System prompts for the 2FA code.

  3. **Delivery/Generation:**

     * **For SMS/WhatsApp/Email:** The user receives a message with the code.

     * **For Authenticator Apps:** The user opens their app (Google/Microsoft Auth) to view the current rotating code.

  4. **Access Granted:** Agent enters the code to establish the session.




### 4.2. First-Time Setup (Registration)

When a user logs in for the first time after 2FA is enabled:

  * **If Channel is SMS/WhatsApp/Email:** The user is prompted to enter their phone number or email address and verify it with a test code and store the user identifier (phone number/email) for future seamless login flow.

  * **If Channel is Authenticator App:** The system displays a **QR Code**. The user scans this code with their phone's Authenticator App to instantly pair the device. User secret stored in IAM for future seamless login flow.

## 5\. Business Rules & Constraints

**Feature**| **Business Rule**  
---|---  
**Channel Exclusivity**|  A tenant configures **one** primary method (e.g., everyone uses WhatsApp). This simplifies helpdesk support and training.  
**Offline Capability**| **Authenticator Apps** are the only method that works without an active internet or cellular connection.  
**Session Enforcement**|  Verification is required for every new login session. "Trusted Device" (long-term persistence) is currently planned for future along with tracking total re-attempts and lockup mechaism in case of max re-attempts.  
**Cost Management**|  Tenants sensitive to telecom costs should prioritize **Authenticator Apps** or **Email** over SMS/WhatsApp.
